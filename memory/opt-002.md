## 教训
- **BASH_SOURCE 与 symlink**: 当脚本经 symlink 调用时, `${BASH_SOURCE[0]}` 是 symlink 路径而非真实路径,
  `dirname`+`cd`+`pwd` 解析到 symlink 所在目录而非脚本真实目录。必须用 `readlink -f` 先解析物理路径。
  这个 bug 让 opt-001 安装的 pre-commit hook 一直没真正生效——提交时其实都跳过了 health check。
  **后续凡涉及"脚本自定位"的代码, 都要考虑 symlink 场景。**
- **WARN 与 FAIL 分离**: health-check 的 FAIL 项 (1-6) 决定 exit code 和崩溃标记, WARN 项 (7-8) 只打印警告。
  这个分离让磁盘/日志等"软"指标能纳入检测而不误触发 safe-mode 回滚。测试要专门覆盖
  "WARN 触发但 exit 0" 这个契约, 否则 WARN 项一旦写错 ERRORS 计数就会悄悄变成 FAIL。
- **测试构造副本的 sed 技巧**: 用 sed 把真实脚本复制成"必失败副本"/"恒成功副本"/"必 WARN 副本",
  避免改真实脚本。注意 sed 替换多行 python 块很麻烦——改单行 `assert` 比替换整个 `python3 -c` 安全。
- **隔离测试不污染真实仓库**: safe-mode 的 stash/reset 逻辑用 mktemp 临时 git 仓库 + sed 改写 PROJECT_DIR 测试,
  绝不在真实仓库造脏改动跑 reset --hard (会丢工作)。
- **范围扩展记录**: pre-commit-hook.sh 不在 opt-002 brief 的"可修改文件"列表里, 但用户明确要求修复,
  属于本轮直接指令优先于 brief 的范围约束。已在 memory 记录, 后续若再动需重新确认。
